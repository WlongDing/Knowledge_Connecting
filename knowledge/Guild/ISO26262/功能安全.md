# 功能安全认证笔记

## 1. 安全机制概述

### 1.1 Power SM (电源安全机制)
- 工作在5V和3.3V下的电压检测安全机制
- 出现高压或低压情况触发LVR、LVD、HVD信号给RCM产生复位或中断

### 1.2 Clock SM (时钟安全机制)

#### FOSC (40M)
- SCG内LOC（Loss of clock）检查
- CMU检测（超频和丢失检测：loc/mis）

#### FIRC (96M)
- AutoTrim校准
- CMU检测（loc/mis）

#### SOSC (32k)
- LOCh和CMU检测

#### PLL (90-150M)
- LOC和LOL检测

#### SIRC (12M)
- AutoTrim自校准
- CMU检测

#### DIVBUS
- LOC检测

### 1.3 FLASH安全机制
- ECC检测：通过ERM寄存器或FLASH API获取ECC错误类型和出错地址
- WDOG监测写或擦完成，检测每一位传输

### 1.4 System RAM安全机制
- ECC保护及注入
- ECC错误反映在ERM模块和MCM模块中

## 3. 认证阶段与流程

### 3.1 第一阶段：功能安全管理
- **安全计划** (Safety Plan) - 需编写，有模板
- **验证计划** (Verify Plan) - 需编写，有模板
- **SFA** (功能安全假设文件) - 将自身当黑盒说明，自身做了什么，外部支撑做什么
- **SEooC** (独立安全单元) - 独立文件
- **软件架构文件** (软件概要设计文件) - 对MCAL内部运行方式进行整理
- MCAL功能级别安全分析/DFA

### 3.2 第二阶段：详细设计与测试
- MCAL白盒测试报告
- 语句分支覆盖率、MC/DC覆盖率100% (ASIL-D)
- Coding Guideline
- Qualification Report of Software Tool
- 软件详细设计
- 软件单元测试
- User Manual
- FIT测试（莱茵提供模板，输出测试报告和测试用例）

### 3.3 第三阶段：送审与颁证
- 所有文件送德国审核
- 颁证，莱茵官网可查

## 5. FIT测试流程

1. 客户基于TUV提供的FIT模板编写FIT测试用例
2. TUV审查测试用例的完整性和正确性
3. 客户开始内部FIT测试（约2-3周）
4. 记录并编写内部FIT报告
5. TUV和客户现场合作完成测试
   - 客户工程师负责测试
   - TUV工程师负责见证测试、记录结果、填写结论
6. 编制正式TUV侧FIT报告（约1周）

## 8. 常见问题与修改反馈

### 8.1 文档格式问题
- SRS文档需要统一格式与封面
- 所有属性名称不能有空格
- 需求文档需要有章节支持
- 外审文件要保证唯一性

### 8.2 需求文档问题
- should（可选）改为shall（必须）
- 用safety不要用security
- 需求comments和remarks需要统一，在comments里写拒绝理由
- 部分实现的需求可以不写在SW Requirements上吗？需要加入EcuM

### 8.3 架构设计问题
- 大白话太多，需要用图描述
- 功能描述用图描述，静态设计和动态设计方法
- 分类画图：独立模块和非独立模块
- 内部静态架构：interface、硬件适配层（分三层：reg, HWA, API）
- 安全机制加图
- 模块之间的接口说明
- 明确模块独立性：独立运行 vs 非独立运行

### 8.4 Coding Guideline问题
- 加上圈复杂度的guideline
- 删掉一些缩写
- 确保代码实施与guideline描述一致
- MCAL模块参数命名规范不统一，与Coding Guideline不吻合

### 8.5 模块设计问题
- 以element（模块）为单位介绍，不要加入driver分层
- 7.1.2明确element颗粒度，直接明确MCAL颗粒度
- 讲明模块是否有用到其他模块功能
- 模块之间的接口（如I2C使用DMA传输时，使用DMA模块提供的接口）

### 8.6 Safety State相关
- 模块内所有提供给上层的接口都拥有故障反馈功能
- 通过反馈用户可进入自定义安全状态

## 12. 其他注意事项

### 架构设计要求
- 架构设计等于FSR + 自己的内部需求
- ISO26262-6的7.5.1：Software architectural design specification
- 静态方面：描述软件组件间的接口
- 动态方面：进程顺序和时序行为（组件启动顺序、函数调用）
- 需提供ASIL表示满足安全需求的证据

### 设计文档结构
- 在module design前面简述时，每个MCAL模块设计需follow AUTOSAR界面设计规定的需求
- 自己添加ID，每个模块安全机制添加ID
- 将FSA中的需求编号对应到Design中，Design自己也要定编号

### ASIL等级
- MCAL模块无法分解，统一ASIL B

### 工具相关
- TCL1不需要证据
- EB生成的代码会完整走一遍ISO验证和validation流程
